# m3l5 — Key Takeaways

- Diagnostyka to synteza z **wielu źródeł**, nie czytanie kodu w ciemno. Sentry → lokalizacja, runtime logs → skala, Playwright (`browser_console_messages` + `browser_network_requests` + `browser_network_request`) → kontekst klient/serwer, kod → root cause. Nawet gdy „bug i tak siedzi w X.ts", trzymaj się procesu — ćwiczysz go pod te przypadki, gdzie nie wiesz nawet, który plik otworzyć.
- **Połknięty błąd** (swallowed error, OWASP A10:2025) to klasa, której proaktywny pipeline nie widzi: try/catch loguje warning, zwraca 200 z wartością policzoną w pamięci, frontend patrzy na status. Sentry zobaczy to dopiero z `captureConsoleIntegration({ levels: ['warn', 'error'] })` — bez tego cichy `console.warn` nigdzie nie trafia. Włącz to w SDK od początku, prune levelów dopiero gdy wzrost ruchu zaczyna palić limit.
- Test-driven bugfixing: **napisz padający test reprodukujący buga zanim naprawisz**. Test sprawdza **zapisany stan** (np. wiersz w `review_states`), nie odpowiedź API (która kłamie 200). Promptem każ agentowi padać z właściwego powodu: „assert that persisted row's `due` moved into the future". Test zostaje w repo jako regression guard. `/10x-tdd` dla unit/integration, `/10x-e2e` dla bugów widocznych dopiero w UI.
- VLM (vision model) używasz w **debugowaniu**, nie w testach — to inna funkcja niż w M3L4. W testach asercja jest wąska („czy ten element wygląda OK"). W debugowaniu zadajesz **pytanie otwarte** („co tu jest nie tak"), bo nie wiesz jeszcze, co się zepsuło. Trzy kategorie: frontier (jakość), Gemini Flash (rozsądny default), Qwen-VL (lokalnie, darmowe). Pamiętaj: spatial bias (przesuwa kursor w stronę środka) i halucynacje — false negative jest groźniejszy niż false positive.
- Weryfikacja fixu = **lustrzane odbicie dochodzenia**. Każde źródło, które pokazało problem, musi pokazać brak: Sentry → no new issues; logi → bez `update_failed`; reprodukcja → karta znika z kolejki; test integracyjny → zielony. Pominięcie warstwy = brak pewności, że fix jest kompleksowy.
