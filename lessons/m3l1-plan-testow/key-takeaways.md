# m3l1 — Key Takeaways

- Mapa ryzyk poprzedza testy. „Write tests for this file" → agent idzie po linii najmniejszego oporu (helpery, gettery, formatery), coverage rośnie, krytyczny flow nadal goły. Punkt wejścia to **scenariusz biznesowy + impact × likelihood**, nie plik. 3-stopniowa skala (wysoki/średni/niski) wystarczy — chodzi o zgodność dwóch osób patrzących na ten sam wiersz, nie precyzję drugiego miejsca po przecinku.
- Jedno z trzech pytań wywiadu warto wynieść osobno: „na co **NIE** chciałbyś, żeby poszedł budżet testowy?". Odpowiedź trafia do sekcji wykluczeń. Za pół roku, gdy ktoś zapyta „a może snapshoty stron marketingowych?", masz spisaną decyzję z powodem.
- Bezpieczeństwo dorzuć ręcznie, agent rzadko zaproponuje sam. Cztery klasy do pokrycia: **autoryzacja (IDOR — czy user A dosięgnie zasobu B przez podmianę ID)**, walidacja inputu, sekrety w logach/bundle, nadużycie zasobów (rate-limit, kosztowne pętle, masowe magic-linki). Brak choćby jednego wiersza security na mapie ≠ "jest bezpiecznie", tylko "nikt nie zapytał".
- Test-plan to **sygnał**, nie diagnoza. Wpis „brak retry w `session.ts:42`" skill odbije: „to opis implementacji, przeformułuj na to, co poczuje użytkownik". Konkrety w kodzie należą do `/10x-research` w fazie rollout'u — jeśli plan udawałby ekspertyzę kodu, zamknąłby research w fałszywej hipotezie.
- Cookbook (sekcja 6 test-planu) startuje jako `TBD — see §3 Phase 1` i to jest celowe. Skill nie udaje, że zna lokalne wzorce, zanim implementacja pokaże, jak projekt naprawdę testuje. Pisz go po fazie, nie przed — kolejny agent (twój za miesiąc) otwiera cookbook z `Lokalizacja:`, `Polityka mockowania:`, `Referencyjny test:`, `Komenda lokalna:` zamiast zgadywać.
