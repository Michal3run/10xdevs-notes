# m3l3 — Key Takeaways

- Hooki to warstwa **deterministyczna** — przetrwają kompresję kontekstu, zmianę instrukcji systemowych, „zapomnienie" przez model. Reguła w `CLAUDE.md` może zostać skompresowana w długiej sesji; hook odpali się zawsze, bo działa poza modelem. Stąd gradacja: szybkie sprawdzenia → hook, miękkie polityki → AGENTS.md.
- Heurystyka warstwy: jeśli check trwa **dłużej niż kilka sekund**, zejdź wyżej (per-edit → pre-commit → pre-push → CI). Lint/format/typecheck w małym repo: per-edit OK. Pełny zestaw testów: pre-commit/pre-push. Jak hook zaczyna cię irytować, to nie problem agenta — sprawdzenie powinno zmienić warstwę albo zawęzić zakres plików.
- Exit code 2 = blokujący feedback do agenta. Inne ≠ 0 logowane, nie blokują. Trywialne korekty (format, brakujący import, zły typ) agent ogarnie sam w następnej iteracji. **Logikę biznesową diagnozuje pozornie** ("coś jest nie tak" + naiwna próba), więc na padający test integracyjny nie odpalaj kolejnego prompta — utwórz osobny change przez `/10x-new` → `/10x-research` → `/10x-plan`.
- Vitest: `vitest related <plik> --run` to **subkomenda, nie flaga** `--related`. `--run` zapobiega trybowi watch (bez tego hook nie zwróci exit code). Od Vitest 4.1 ustaw `AI_AGENT=1` — kompaktowy output dla hooków, mniej tokenów w kontekście. Parsowanie ścieżki przez `jq -r .tool_input.file_path` na stdin (Codex/Cursor mogą mieć inne pola).
- Codex ma **trust model po hashu**: hooki z repo (`.codex/hooks.json`) wymagają zatwierdzenia komendą `/hooks` i każda zmiana wymaga ponownego potwierdzenia. „User-level w `~/.codex/config.toml` działa, projektowe nie" to nie bug, tylko świadoma bariera. Copilot w VS Code parsuje matchery z formatu Claude, ale ich **nie egzekwuje**, używa camelCase (`tool_input.filePath`) i innych nazw narzędzi (`create_file` zamiast `Write`).
