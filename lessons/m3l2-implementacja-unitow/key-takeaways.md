# m3l2 — Key Takeaways

- Coverage słabo koreluje z wykrywaniem błędów. Agent rzadko zostawia kod **niepokryty** — częściej zostawia go **pokrytym, ale nieobronionym**. Stryker (mutation testing) pokazuje tę różnicę: 50 zielonych testów w 10xCards po dwóch fazach test-planu = ~38% celowych błędów przeszłoby niezauważenie. Mutation score w mianowniku ma `killed + survived + no-coverage`, w liczniku `killed + timeout`.
- Wyrocznia nie może pochodzić z testowanego kodu. Funkcja `getNextInterval(prev, grade)` zwraca `prev * grade`, więc dla `grade=0` zwróci `0` — agent zapisze asercję `toBe(0)`. Reguła biznesowa mówi reset do 1 dnia. Test zabetonował buga. Daj agentowi progi/założenia z PRD: „rabat 10% >200zł, 15% >500zł" — nie „napisz testy do tej funkcji".
- Sprawdź, czy test **potrafi zawieść**. Zepsuj na chwilę kod produkcyjny i zobacz, czy test pada. Jeśli dalej zielony — asercja niczego nie pilnuje. To uproszczona wersja mutation testingu, którą odpalasz ręcznie po każdej fazie. Stryker (`npx stryker run` z `--mutate "src/srs.ts:1-40"`) automatyzuje to dla obszarów z wysokiego ryzyka — nie traktuj jednak mutation score jak coverage do gonienia w 100%, bo wtedy zaczniesz pisać kruche mirrory implementacji.
- Research **może zmienić cel zadania**, nie tylko go uzupełnić. Case study z 10xCards: ryzyko „częściowy commit przy błędzie" — research odkrył, że rollbacku nie ma, `saveSession()` zwraca `ok:true` mimo `orphan_review_state`. Decyzja: **nie pisz testu na rollback, którego nie ma**. Przypnij obecny, świadomie tolerowany kontrakt + wynotuj otwarte ryzyko produktowe.
- `/10x-tdd` jako alternatywa dla `/10x-implement` w wybranej fazie — gdy umiesz nazwać pierwszy czerwony test jednym zdaniem (walidatory, parsery, kontrakty API, reducery, bugfixy z reprodukcją). Mieszaj oba w jednym `plan.md` (wspólny `## Progress`); skill zakłada, że runner i konfiguracja już istnieją — bootstrap testów to `/10x-implement`.
