# m3l2 — Linki

## Tools

- [Stryker JS](https://stryker-mutator.io/docs/stryker-js/introduction/) — mutation testing dla JS/TS; `npx stryker init` + `stryker run`. Tryb `--incremental` i `--mutate "path:start-end"` do trzymania czasu w ryzach.
- [Vitest CLI](https://vitest.dev/guide/cli.html) — `--changed`, `--coverage`, integracja ze Strykerem przez `@stryker-mutator/vitest-runner`.

## Reference

- [Benchmarking LLMs for Unit Test Generation from Real-World Functions (Huang et al., ACM TOSEM 2026)](https://arxiv.org/abs/2508.00408) — pokrycie spada z ~92% na TestEval do ~45% linii / ~30% gałęzi na realnym kodzie. Twardy dowód, że benchmark ≠ projekt.
- [LLMs for Unit Test Generation: Achievements, Challenges, Opportunities (Chu et al., 2025)](https://arxiv.org/abs/2511.21382) — przegląd: oracle problem i słaba korelacja coverage z error detection wymuszają pętlę z feedbackiem.
- [Quality Assessment of Python Tests Generated by LLMs (Alves et al., 2025)](https://arxiv.org/abs/2506.14297) — błędne asercje to najczęstszy błąd w testach od LLM-ów.
- [TestForge: Feedback-Driven, Agentic Test Suite Generation](https://arxiv.org/abs/2503.14713) — przykład pętli „test → uruchomienie → poprawka asercji" jako antidotum na zielony mirror.
