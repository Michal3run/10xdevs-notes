# m3l4 — Key Takeaways

- Co pokażesz w `seed.spec.ts`, to dostaniesz w generowanych testach. Generator Playwrighta używa seed jako wzorca dla **wszystkich** kolejnych testów. Jeśli seed ma `getByRole`, generuje `getByRole`; jeśli ma `page.waitForTimeout(2000)`, powiela ten antywzorzec wszędzie. Seed musi też pokazywać unikalne identyfikatory (`Date.now()` w nazwie) i pełny cykl setup → akcja → asercja → cleanup w jednym teście.
- VERIFY ≠ zielony test. Po uruchomieniu na zielono **odwróć w kodzie produkcyjnym chronione zachowanie** (faktyczne psucie ryzyka, nie kosmetyka) i potwierdź, że test pada. Jeśli zostaje zielony — asercja niczego nie pilnuje, wracaj do GENERATE. Cofnij zepsucie natychmiast po sprawdzeniu, **nigdy nie commituj**.
- Bramka kwalifikacji E2E: ryzyko musi przechodzić przez **wiele granic systemu** (auth + routing + API + baza) albo istnieć tylko w wyrenderowanym UI. Jeśli izolowana funkcja by skłamała — E2E. Jeśli unit/integracja by wystarczyła — wracaj do `/10x-tdd`/`/10x-implement`. E2E jest najwolniejsza i najbardziej krucha; sięgasz po nią ostatnią.
- Mockowanie na granicy sieci, nie modułów. Wewnętrzne granice (auth, routing, baza) **prawdziwe** — to w nich kryją się ryzyka integracyjne. Drogie/niedeterministyczne zewnętrzne API (LLM, bramki płatności) mockuj na warstwie HTTP (np. MSW dla OpenRoutera). To pragmatyczny wzorzec, nie wyjątek.
- Healer (Playwright Test Agents) **pomaga przy zmianie selektora**, **szkodzi przy zmianie logiki biznesowej**. Gdy backend zmienia format odpowiedzi, healer dopasuje asercję do nowego (błędnego) zachowania → maskuje buga. Granica: zmiana w UI = healer; zmiana w kontrakcie = ręczna analiza, nie healer.
